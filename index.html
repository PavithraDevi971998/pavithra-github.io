    5. Use double quotes to all attribute values
  • Best Practices

    2. Use semantically appropriate elements to layout your page

CSS 3

  • Coding Standards
    1. Use tabs, not spaces, to indent each property
    2. Each selector should be on its own line, ending in either a comma or an opening curly brace.
    3. use lowercase and separate words with hyphens when naming selectors. Avoid camelcase and underscores
    4. Use human readable selectors that describe what element(s) they style
    5. Attribute selectors should use double quotes around values
    6. Properties should be followed by a colon and a space
    7. All properties and values should be lowercase, except for font names and vendor-specific properties
    8. Space before the value, after the colon
    9. Always end value in a semicolon
    10. Use double quotes rather than single quotes, and only when needed, such as when a font name has a space or for the values of the content property.
    11. Rule sets for media queries should be indented one level in
    Best Practices
    1. Refrain from using over-qualified selectors, div.container can simply be stated as .container
    2. Use hex code for colors, or rgba() if opacity is needed. Avoid RGB format and uppercase, and shorten values when possible: #fff instead of #FFFFFF
    3. keep media queries grouped by media at the bottom of the stylesheet
    4. Comment, and comment liberally.
    5. DOM will change over time, target the element you want to use as opposed to “finding it” through its parents. Example: Use .highlight on the element as opposed to .highlight a (where the selector is on the parent)

JavaScript

  • Coding Standards
    1. Indentation with tabs
    2. No whitespace at the end of line or on blank lines
    3. Lines should usually be no longer than 80 characters, and should not exceed 100 (counting tabs as 4 spaces).
    4. Any , and ; must not have preceding space
    5. There should be a new line at the end of each file
    6. Objects and arrays can be declared on a single line if they are short
    7. if, else, for, while, and try blocks should always use braces, and always go on multiple lines.
    8. When a statement is too long to fit on one line, line breaks must occur after an operator.
    9. Comments come before the code to which they refer, and should always be preceded by a blank line.
    10. Use single-quotes for string literals
    11. usage of switch statements is generally discouraged, but can be useful when there are a large number of cases
  • Best Practices
    1. Creating arrays in JavaScript should be done using the shorthand [] constructor rather than the new Array() notation.
    2. When iterating over a large collection using a for loop, it is recommended to store the loop’s max value as a variable rather than re-computing the maximum every time
